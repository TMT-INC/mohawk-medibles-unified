# Unified Site Audit — 2026-06-23 (pre-July-1 cutover)

Consolidated, deduplicated triage of two verifier-checked audit runs (batch-1: auth-session + pay-wampum; batch-2: 9 remaining dimensions) over the `mohawk-medibles-unified` Next.js codebase that will replace the live WooCommerce store at DNS cutover (target July 1, 2026). After merging duplicates across both files there are **43 confirmed findings** (verifier-adjusted severities): **11 high, 16 medium, 13 low, 3 info**. Classified against cutover impact, **17 are July-1 BLOCKERS** (expose customer PII/money, break a core buy/login/checkout/account flow, allow order/payment state to be forged or mis-set, or are a data-integrity/regulatory-termination risk at launch) and **26 are POST-LAUNCH** (defense-in-depth not exploitable pre-flip, SEO/a11y/CRO polish, notification reliability, minor data hygiene, cosmetic). Several findings were merged: the `x-forwarded-for` leftmost-hop spoof appears in 4 routes, the spoofable `x-user-*` header trust spans reviews + gift-cards, the un-awaited-email/SMS pattern spans 4 handlers, and the gift-card weaknesses cluster on the same two routes. 4 findings were rejected as non-issues.

## July-1 Blockers

Ranked critical-impact → low. Each blocker either leaks customer PII at scale, lets money/order/payment state be forged or set incorrectly, breaks a core customer flow, or is a launch-day data-integrity / regulatory exposure.

### 1. Public order-tracking endpoint is an enumerable full-PII IDOR [high]
- **Files:** `app/api/orders/track/route.ts:34-148` (sequential `MM-{wcId}` set by `app/api/webhooks/woocommerce/route.ts:85`, `app/api/sync/orders/route.ts:77`, `app/api/cron/wc-sync/route.ts:72`, `scripts/full_order_sync.ts:61`)
- **Root cause:** GET `?orderNumber=MM-N` is not in `middleware.ts` PROTECTED_PATHS and does no ownership/email/session check; it returns firstName, lastName, street1, city, province, postalCode, line items, total, status history, tracking. WC-synced order numbers are sequential.
- **Exploit/impact:** Walk MM-1..MM-N to scrape full name + home address + cannabis purchase history of all ~25,918 imported customers; the only barrier is an in-memory 30/min limiter keyed on the spoofable `x-forwarded-for` (see #4).
- **Fix:** Require proof of ownership — gate behind the `mm-session` cookie and verify `order.userId === session.userId` (as `app/api/account` does), OR require a non-enumerable secret factor (billing email match or a random per-order tracking token). Additionally drop `street1`/`lastName` from any unauthenticated response (return city/province/status/tracking only).

### 2. Spoofable `x-user-id` / `x-user-role` trusted on non-protected routes — gift-card theft + cross-user impersonation [high]
- **Files:** `app/api/reviews/route.ts:27-28,91-93`; `app/api/gift-cards/route.ts:14` (GET). Root cause in `middleware.ts:131-138,190-192`. (Secure pattern to copy: `app/api/flash-sales/route.ts:52`.)
- **Root cause:** Middleware only SETS `x-user-*` on PROTECTED_PATHS and never STRIPS inbound copies; non-protected routes get `NextResponse.next()` forwarding the client's original headers verbatim. These routes trust those headers as the authenticated identity/role.
- **Exploit/impact:** `curl -H 'x-user-id: <victim-uuid>' /api/gift-cards` returns the victim's gift-card **code + currentBalance + transactions** (directly monetizable theft, no auth). `-H 'x-user-id: <victim>' POST /api/reviews` impersonates any user; `-H 'x-user-role: ADMIN'` bypasses review moderation.
- **Fix:** In `middleware.ts`, on EVERY request (including the non-protected early-return), clone request headers and `delete` `x-user-id`/`x-user-role`/`x-user-email` before forwarding so only middleware-injected values survive. Additionally have reviews + gift-cards routes derive identity by verifying the `mm-session` JWT directly (`verifySessionToken`, like flash-sales) and ignore the headers.

### 3. Coupon applied with zero validation + percentage stacking via repeated codes [high]
- **Files:** `app/api/checkout/route.ts:120-135` (schema fields ignored: `prisma/schema.prisma:444-465`)
- **Root cause:** The coupon loop only filters `{ code, active:true }`; it never checks `validFrom`/`validUntil`, `minOrderTotal`, `maxUses`/`usedCount`, or `perCustomer`, never increments `usedCount`, never writes a `CouponUsage` row, and never de-dups `coupon_codes`. PERCENTAGE discount is added per array entry against raw subtotal.
- **Exploit/impact:** Any active/leaked/expired/one-time code is reusable forever and ignores min-spend; `["SAVE50","SAVE50","SAVE50"]` yields 150%-of-subtotal discount, flooring the order to shipping-only. Direct, automatable margin loss.
- **Fix:** In the loop: (a) de-dup `coupon_codes` (ideally cap to 1 coupon); (b) reject if `validFrom>now`, `validUntil<now`, `subtotal<minOrderTotal`, or `maxUses!=null && usedCount>=maxUses`; (c) enforce `perCustomer` via a `CouponUsage` count for the user; (d) compute percentage discount once against subtotal, never additively per duplicate; (e) cap total discount at subtotal; (f) inside the existing `$transaction`, atomically `updateMany` `usedCount` with a `usedCount<maxUses` guard and create a `CouponUsage` row, failing the order if the claim doesn't land.

### 4. IP rate-limit / IP allowlist trust the leftmost (client-supplied) `x-forwarded-for` hop [high]
- **Files (merged — same root cause, 4 sites):** `app/api/webhooks/digipay/route.ts:22-30` + `lib/digipay.ts:133-137` (security gate — high impact); `lib/rateLimit.ts:188-194` (auth/checkout throttle); `app/api/orders/track/route.ts:35` (PII limiter, see #1); inconsistency reference `lib/captcha.ts:70-77` (correctly uses cf-connecting-ip first).
- **Root cause:** Code takes `x-forwarded-for.split(",")[0]` — the leftmost, attacker-injected entry — instead of the platform-set `cf-connecting-ip` / Vercel's `x-vercel-forwarded-for`. Cloudflare/Vercel APPEND the real IP rather than replacing, so the forged left hop wins.
- **Exploit/impact:** The Digipay postback (which flips an order to PROCESSING + paymentStatus PAID and auto-creates a ShipStation label) is authenticated ONLY by this IP allowlist (no HMAC, unlike BTCPay). An attacker places their own order, learns the exact total, then POSTs `{session:<orderId>, amount:<total>}` with `x-forwarded-for: 185.240.29.227` → free physical-goods shipment. The same spoof gives each request a fresh rate-limit bucket, defeating the 5/min auth throttle (brute-force / credential stuffing) and the order-track PII limiter.
- **Fix:** (a) **Digipay**: add a real Digipay shared-secret/HMAC verification over the raw postback body with `timingSafeEqual` before any DB mutation (the project already references `DIGIPAY_POSTBACK_TOKEN`); treat IP allowlist as defense-in-depth only. (b) **All IP keying** (`getClientIP`, digipay, order-track): prefer `cf-connecting-ip` / `x-vercel-forwarded-for` (platform-set, non-forgeable), and never the leftmost `x-forwarded-for`. (c) Add a secondary rate-limit key on the submitted email for login/reset.

### 5. Gift-card redeem is unauthenticated (CSRF-only) — drain any card + phantom redemptions against arbitrary orderIds [high]
- **Files:** `app/api/gift-cards/redeem/route.ts:9-70` (middleware fall-through `middleware.ts:131-138`; checkout never reads gift cards)
- **Root cause:** Route isn't in PROTECTED_PATHS, so no session check; its only guard is `verifyCsrf` (double-submit cookie), which a scripted/non-browser caller trivially satisfies by sending an identical cookie+header pair. It looks the card up purely by attacker-supplied `code` (no ownership check vs `purchasedByUserId`/`recipientEmail`), debits an attacker-supplied `amount`, and writes a redemption tied to an unvalidated `orderId`. Read-then-write (no atomic guard) is also race-able.
- **Exploit/impact:** Given any known card code, zero a stranger's balance and stamp phantom redemptions on arbitrary orders; over-redeem via the race.
- **Fix:** Require an authenticated session (add `/api/gift-cards` to PROTECTED_PATHS with a CUSTOMER-allowed tier, or `requireUser` in-route); enforce card ownership/recipient match; bind redemption to a server-resolved order owned by the caller and validate `orderId` is a real unpaid order whose remaining total ≥ amount; make the debit atomic: `updateMany({ where:{ id, currentBalance:{ gte: amount } }, data:{ currentBalance:{ decrement: amount } } })` and reject if `count!==1`; wire gift-card application into the server-side checkout total instead of treating redeem as a free-floating mutator.

### 6. WooCommerce webhook signature verification fails OPEN when `WC_WEBHOOK_SECRET` is unset/empty [high]
- **Files:** `app/api/webhooks/woocommerce/route.ts:398-405` (PUBLIC path `middleware.ts:44`; `.env.example:13` ships it empty + "Optional")
- **Root cause:** HMAC check is wrapped in `if (wcWebhookSecret) {...}`, so an empty/undefined secret (both falsy) skips verification entirely and the handler mutates the DB from the raw body. No middleware gate.
- **Exploit/impact:** With the secret blank, anyone POSTs `resource=order` `{id:12345,status:"processing"}` → order flipped to PROCESSING + paymentStatus PAID (mapPaymentStatus maps `processing`→PAID) and addresses overwritten; `resource=product` rewrites storefront prices; `resource=coupon` mints a 100%-off code. Latent fail-open trap even if prod currently sets the secret (any future blanking silently disables all auth).
- **Fix:** Fail closed — `if (!wcWebhookSecret) return 503` BEFORE the verify, then run `verifyWebhookSignature` unconditionally and reject requests missing `x-wc-webhook-signature`. Mark the var **required** (not Optional) in `.env.example`.

### 7. Stored XSS via product description through a bypassable regex sanitizer [high]
- **Files:** `app/product/[slug]/ProductDetailClient.tsx:24-29,529`; raw write paths `app/api/webhooks/woocommerce/route.ts:274,301`, `app/api/sync/products/route.ts:93,129`, `app/api/admin/products/route.ts:136,187`
- **Root cause:** `longDescription` is stored raw (unlike `shortDescription`/`metaDescription`, which are tag-stripped) and rendered via `dangerouslySetInnerHTML` through a home-rolled `sanitizeHTML()` that only removes `<script>` and **quoted** `on*=` handlers. Unquoted handlers, `javascript:` URLs, `<svg onload>`, `<img onerror>`, `<iframe>` all survive.
- **Exploit/impact:** A poisoned product description (compromised WC contributor, vulnerable WC plugin, or malicious sync payload) yields persistent stored XSS on the public `/product/[slug]` page for every visitor. (Note: `mm-session` is HttpOnly so the JWT isn't directly readable, but the CSRF token is JS-readable, so an authenticated viewer can be driven to same-origin authenticated actions.)
- **Fix:** Replace `sanitizeHTML` with DOMPurify (`isomorphic-dompurify`) using an allowlist (p, br, strong, em, ul/ol/li, a[href http/https/mailto], h2-h4), stripping all event handlers, `javascript:`/`data:` URLs, and `<script>/<style>/<iframe>/<svg>/<object>/<embed>`. Apply at WRITE time in the woocommerce/sync/admin handlers (as `shortDescription` already is) AND at render time as defense-in-depth.

### 8. `lib/prisma.ts` noop silently returns empty data for the live dispensary directory [high]
- **Files:** `lib/prisma.ts:41-51`; consumers `app/api/dispensaries/route.ts`, `app/api/dispensaries/[slug]/route.ts`, `app/directory/page.tsx`, `app/directory/dispensary/[slug]/page.tsx`
- **Root cause:** `createPrismaClient()` does `new PrismaClient()` with NO adapter, wrapped in try/catch that falls back to a `noopPrisma` Proxy (findMany→[], count→0, create→[]). Verified empirically: installed `@prisma/client` 7.6.0 THROWS on a bare constructor (driver-adapter required), so the catch ALWAYS fires. Only `lib/db.ts` builds the real adapter-backed client.
- **Exploit/impact:** After cutover, `/api/dispensaries` and the `/directory` pages return `{success:true,data:[],total:0}` with HTTP 200 — a permanently empty directory with no error, no Sentry, no log. Admin POST "succeeds" (201, `data:[]`) while writing nothing.
- **Fix:** Delete `lib/prisma.ts` and repoint the four consumers to `import { prisma } from "@/lib/db"`. If a fallback is kept, construct it WITH `new PrismaClient({ adapter: new PrismaPg(new pg.Pool({connectionString: process.env.DATABASE_URL})) })` and RE-THROW real DB errors instead of swallowing them into empty arrays.

### 9. `$0`-priced products are buyable for $0 — no `price>0` guard on detail page or at checkout [high]
- **Files:** `app/api/checkout/route.ts:104-114,137-140`; `app/product/[slug]/page.tsx:55`; `lib/products.ts:169-178,227`; data rows `lib/productData.ts:1322,4531,6750` (ids 33, 112, 168). The grid hides them at `app/api/products/route.ts:51` (`filter(p=>p.price>0)`) but detail/checkout don't.
- **Root cause:** `getProductBySlug`/`getAllProducts` filter only by `status:"ACTIVE"` (no price floor), and checkout computes `lineTotal = product.price * qty` with no lower bound. Three ACTIVE concentrate SKUs (WC variable/price-range products) serialized to `price:0`.
- **Exploit/impact:** Navigate directly to a `$0` SKU (reachable from sitemap/search/recommendations, none of which apply the grid filter), add to cart, check out at `$0` subtotal — pay only $15 shipping (or $0 if cart padded over the $149 free-ship threshold). A real concentrate ships for ~free.
- **Fix:** In `checkout/route.ts` reject `product.price <= 0` (400 "Product not available for purchase"). Exclude `price<=0` from `getAllProducts()`/the detail page (`notFound`). Fix the underlying DB rows so variable/price-range products carry a real base price (sync from WC `prices.price_range.min_amount`).

### 10. Middleware gates ALL `/api/trpc` to admin roles — every customer-facing tRPC feature is dead [high]
- **Files:** `middleware.ts:28-36,177-186` (callers incl. `app/rewards/page.tsx:82-85`, `app/deals/DealsClient.tsx:21`, `app/wishlist/shared/[code]/page.tsx:19`, and sitewide banners `components/ContestBanner.tsx:45`, `DealOfTheDay.tsx:36`, `DealOfTheDayBanner.tsx:39`, `GiftTierProgress.tsx:18`, `GoogleReviewsWidget.tsx:43`, `HappyHourBanner.tsx:50`)
- **Root cause:** `ROLE_REQUIREMENTS["/api/trpc"] = [ADMIN,SUPER_ADMIN,LOGISTICS,SUPPORT]` and `/api/trpc` is in PROTECTED_PATHS, so anonymous visitors get 401 and logged-in CUSTOMERs get 403 — before per-procedure `publicProcedure`/`protectedProcedure` auth ever runs.
- **Exploit/impact:** Functional breakage (fails closed, no leak): rewards/loyalty dashboard, daily deals, happy hour, shared wishlists, compare, suggestions, and every sitewide banner error out on every page for real customers.
- **Fix:** Remove `/api/trpc` from PROTECTED_PATHS and delete the `ROLE_REQUIREMENTS["/api/trpc"]` entry; let per-procedure auth in `server/trpc/trpc.ts` (publicProcedure/protectedProcedure/adminProcedure) be the single source of truth (the route handler already derives role from the cookie). If a coarse edge pre-filter is wanted, gate only specific admin router prefixes, never the whole surface.

### 11. CSRF enforced server-side but no client ever sends `x-csrf-token` — every auth/account POST 403s [high]
- **Files:** `lib/csrf.ts:53-78`; `middleware.ts:88-91`; protected routes `app/api/auth/route.ts:50`, `app/api/account/route.ts:115`, `app/api/gift-cards/route.ts:31`, `app/api/gift-cards/redeem/route.ts:11`, `app/api/reviews/route.ts:86`, `app/api/contact/route.ts:22`, `app/api/newsletter/subscribe/route.ts:16`; client submitters `login/page.tsx:26`, `register/page.tsx:39`, `forgot-password/page.tsx:20`, `reset-password/page.tsx:40`, `account/page.tsx:138-212`, `gift-cards/page.tsx:67`
- **Root cause:** `verifyCsrf` requires BOTH the `mm-csrf` cookie AND a matching `x-csrf-token` header, but the cookie is only seeded by middleware on non-API page responses and NO client reads it / sends the header. Every mutating fetch sends only `Content-Type: application/json`.
- **Exploit/impact:** Functional outage — login, registration, password reset, profile/password/address changes, reviews, contact, gift-card redeem all return 403 by default, forcing the operator to `SKIP_CSRF=true` (which globally disables CSRF). (The cross-site-attack half is blunted by SameSite=lax + JSON-body parsing.)
- **Fix:** Add a `getCsrfToken()` util that reads the non-HttpOnly `mm-csrf` cookie via `document.cookie` and a shared `apiFetch` wrapper that injects `x-csrf-token` on all POST/PUT/DELETE to same-origin `/api` routes; have middleware seed `mm-csrf` before first navigation (e.g. on API responses too) so the cookie exists by the time the client POSTs; verify login/register/reset/account end-to-end return 200 with CSRF enabled. Do NOT ship `SKIP_CSRF=true`.

### 12. Gift-card purchase endpoint mints active, spendable cards with NO payment captured [medium]
- **Files:** `app/api/gift-cards/route.ts:29-95` (live redeem path `app/api/gift-cards/redeem/route.ts`)
- **Root cause:** POST creates a `giftCard` with `status:"active"`, `currentBalance:amount` immediately, gated only by `verifyCsrf` (no auth, no order, no payment, no PENDING state). `purchasedByUserId` comes from the spoofable/absent `x-user-id` header.
- **Exploit/impact:** Anyone POSTs `{amount:500}` and receives a live `MM-` code worth $500 never paid for, immediately redeemable through the existing redeem endpoint; pollutes the financial ledger with unpaid `GiftCardTransaction` rows and emails live codes.
- **Fix:** Create cards in a PENDING/unfunded status that cannot be redeemed; require purchase through the normal paid checkout (an order line item) and flip to `active` only from the payment webhook (Wampum deposit-confirmed / BTCPay settled) after funds settle; require authentication and derive `purchasedByUserId` from the verified session, not a header.

### 13. account update-password parses the wrong hash format — 500s for every user, writes login-incompatible hash [medium]
- **Files:** `app/api/account/route.ts:176-217` (shared helpers `lib/auth.ts:24-29,38-45`)
- **Root cause:** Handler splits the stored hash on `:`, treats salt as base64, and uses SHA-256, but `hashPassword()` produces `pbkdf2$<iter>$<hexSalt>$<hexHash>` (`$`-delimited, hex, SHA-512). For real accounts `split(":")` yields one element → `atob(undefined)` throws; the handler has no try/catch → unhandled 500. On (unreachable) success it writes a colon/base64/SHA-256 hash that `verifyPassword` can never match → permanent lockout. (WC-synced `passwordHash===''` users get a clean 400, unaffected.)
- **Exploit/impact:** The change-password feature is 100% broken for every account with a real password — a core credential-management flow returns 500.
- **Fix:** Replace the bespoke crypto block with the shared helpers: `if (!(await verifyPassword(currentPassword, user.passwordHash))) return 403; const passwordHash = await hashPassword(newPassword);` so storage stays `$`/hex/SHA-512 and round-trips with login; wrap the handler in try/catch returning a generic 500.

### 14. deposit-confirmed marks an order PAID without verifying deposited amount vs order total [medium]
- **Files:** `app/api/webhooks/wampum/deposit-confirmed/route.ts:84-136` (HMAC gate `lib/wampum.ts:44-112`); same trust gap in `transfer-received` and `reopen-order`
- **Root cause:** The claim `updateMany` flips status to PAYMENT_CONFIRMED/PAID gated only on current status; `t.amount` is never compared to `order.total` (used only in email/history strings). The customer email at line 136 asserts the full `order.total` "has been received and confirmed" regardless of what was actually deposited.
- **Exploit/impact:** HMAC-gated, so not directly attacker-reachable — but the store has zero local defense: any Wampum/Blacfin upstream amount-tolerance gap or operator error makes a $1 transfer confirm a $500 order and ships product, with a false "full amount received" email.
- **Fix:** Before the claim `updateMany`, validate: `const paid = Number(t.amount); if (!Number.isFinite(paid) || paid + EPS < Number(order.total)) { record SHORT_PAID, alert ops, return 409 — do NOT mark PAID }`. Persist the actual paid amount and print it (not `order.total`) in the confirmation email. Apply the same expected-total assertion to `reopen-order` before re-decrementing inventory.

### 15. WC webhook handler has no idempotency/recency guard — replayed/older payloads regress paid orders [medium]
- **Files:** `app/api/webhooks/woocommerce/route.ts:96-151` (same delete/recreate pattern in `app/api/sync/orders/route.ts:108-111`, `app/api/cron/wc-sync/route.ts`)
- **Root cause:** `handleOrder` unconditionally re-applies `status`/`paymentStatus` from the WC payload and runs `orderItem.deleteMany`+recreate on every delivery; `date_modified` is never read — no recency/dedupe key. WC re-delivers on 5xx and can arrive out of order. Conflicts with the native e-Transfer rail (Wampum atomically flips the SAME `Order` row to PAID).
- **Exploit/impact:** A re-delivered/out-of-order `order.updated` carrying an earlier on-hold status silently downgrades a Wampum-confirmed PAID/PROCESSING order back to ON_HOLD/PENDING and flips paymentStatus off PAID — happens even without an attacker; an attacker can do it deliberately if the WC secret is unset (#6). (Note: empty `line_items` does NOT wipe items — line 112 guards `if (order && wc.line_items?.length)`.)
- **Fix:** Store WC `date_modified` and skip if the incoming payload is not newer. Do NOT overwrite `paymentStatus` from WC status when the local order was confirmed by the native rail (Wampum/BTCPay). Replace blind `deleteMany`+create with an upsert keyed on WC line-item id. Add a processed-webhook-delivery-id dedupe table to drop replays.

### 16. Products with no Inventory row bypass all stock checks — unlimited oversell, never auto-OUT_OF_STOCK [medium]
- **Files:** `app/api/checkout/route.ts:162-191`; detail-page gap `app/product/[slug]/page.tsx:66-82`; importer that can omit rows `scripts/import_to_db.ts:91-148,160-170`
- **Root cause:** Every stock guard (insufficient-stock throw, decrement, auto OUT_OF_STOCK flip) is nested inside `if (inv)`. The bulk importer creates products ACTIVE but seeds inventory in a separate pass that `continue`s past anything missing from `inventory.tsv`, so ACTIVE products with no Inventory row exist; the only order gate is `status:"ACTIVE"`.
- **Exploit/impact:** Such products can be ordered in any quantity (≤50/line) repeatedly forever; the store oversells stock it can't ship and the SKU never auto-hides. Operational fulfillment risk at launch.
- **Fix:** Treat a missing Inventory row as hard-fail "out of stock" for physical products, OR explicitly create inventory rows for every ACTIVE product during sync so the existing guards always run. At minimum, when `inv` is null, flag the order for manual stock verification instead of silently accepting unlimited quantity.

### 17. Shop-by-Mood "Relief" funnel makes Cannabis Act s17 prohibited therapeutic claims [high]
- **Files:** `lib/intentMapping.ts:76,118-124`; `app/shop-by-mood/page.tsx` → `ShopByMoodClient.tsx:54,99,102-116`; `components/ShopByMood.tsx:78`. Closely related: site-wide FAQPage JSON-LD medical claims `lib/seo/aeo.ts:61-62,68-69` rendered via `app/layout.tsx:171` (entry 10 / line 101 is excluded by the `slice(0,8)`).
- **Root cause:** A recreational store publicly markets therapeutic/health benefits — "Soothe What Hurts", "Therapeutic CBD blends ... crafted for pain management, recovery, and healing comfort", benefit pills (Pain management, Anti-inflammation, Muscle recovery, Therapeutic comfort), plus relax/sleep "Anxiety ease"/"Stress relief"/"Insomnia relief", and RELIEF_KEYWORDS (pain, arthritis, anxiety, migraine, nausea, medicinal, healing). These render publicly and are emitted as machine-readable FAQPage structured data on every route.
- **Exploit/impact:** Cannabis Act / Cannabis Regulations promotion prohibitions on health/therapeutic claims by a non-medical (recreational) licensee — Health Canada enforcement exposure (and indexable JSON-LD makes it worse). Treated as a launch blocker because it is a legal/regulatory liability that can attract enforcement and jeopardize the storefront at go-live.
- **Fix:** Remove/rename the "Relief" mood and strip disease/therapeutic keywords from `intentMapping.ts` (line 76 description + RELIEF_KEYWORDS); rewrite `ShopByMoodClient.tsx` relief/relax/sleep copy and benefit pills to neutral, non-therapeutic language; scrub the therapeutic claims from `MASTER_FAQ_DATABASE` entries 5 & 6 (aeo.ts:61-62, 68-69) or exclude any health-claim FAQ from `getFAQsForSchema()`. Have counsel review before launch.

## Post-launch backlog

Defense-in-depth not exploitable pre-flip, notification reliability, SEO/a11y/CRO, minor hygiene. One bullet each.

- **[medium] Public checkout-verify leaks order financials by sequential WC id** — `app/api/checkout/verify/route.ts:12-61`. No ownership/`key` check (the documented `key` param is never read); enumerable `?order=1..N` dumps total/paymentMethod/status/item names+prices (NOT identity PII). Fix: require + constant-time-compare an unguessable per-order `key`/token, drop the `orderNumber:{contains}` substring fallback for exact match.
- **[medium] deal-rotation cron fails OPEN when `CRON_SECRET` unset** — `app/api/cron/deal-rotation/route.ts:27-34`. Returns `true` if no secret (the other 5 crons return `false`); `.env.example:43` ships it empty. If blank in prod, anyone forces ~30%-off deals + un-features legit deals (idempotency-bounded to one per window). Fix: match siblings — `if (!cronSecret) return false`.
- **[medium] Sync routes reuse `AUTH_SECRET` as a static bearer + non-constant-time compare** — `app/api/sync/{orders,products,customers}/route.ts` (POST 167-170, GET 261-262). `x-sync-secret` equals the JWT signing key, so a leaked header is a SUPER_ADMIN session-forgery key. Fix: dedicated `SYNC_SECRET` distinct from `AUTH_SECRET`, compared with `crypto.timingSafeEqual`.
- **[medium] BTCPay confirmation/ops emails fired un-awaited — dropped on Vercel lambda freeze** — `app/api/webhooks/btcpay/route.ts:98,109,141-156,158`. Order IS reliably persisted PAID; only the 4 detached `sendEmail` calls are unreliable (sibling wampum/deposit-confirmed correctly awaits). Fix: `import { after } from "next/server"` and wrap each send.
- **[medium] Auth welcome + password-reset emails fired un-awaited** — `app/api/auth/route.ts:40-42,126-128`. Reset email is the only account-claim path for synced/guest users; the token IS persisted (retry-able) but the Resend fetch races the freeze. Fix: `await`/`after()` the sends (prefer awaiting the reset email so hard failures surface).
- **[medium] Shop listing API serves stale hardcoded `productData.ts`, not the DB — split-brain catalog** — `app/api/products/route.ts:2,51-96` (frozen 2026-02-26 snapshot) vs detail/sitemap/checkout reading the DB.
